# 操作和按钮逻辑

## 维护规则

这份文档记录手机 app 的操作入口、按钮含义和扫码状态机。后续每次改扫码、按钮、导航或写入流程，都要同步更新这里，避免代码逻辑和产品理解分叉。

## 当前版本快照：0.1.4-debug

主导航保留四个入口：

```text
快捷扫码 / 查库存 / 新增 / 流水
```

### 快捷扫码

快捷扫码是日常主入口，打开后围绕相机扫码工作。

当前按钮：

| 按钮 | 作用 |
|---|---|
| 开始扫码 | 打开后置相机，开始识别二维码 |
| 停止 | 停止相机预览，释放摄像头 |
| 打开手电筒 / 关闭手电筒 | 相机启动后出现，设备支持 torch 时可用 |
| 盘点称重 | 手动进入称重盘点上下文 |
| 绑定库位 | 手动进入库位绑定上下文 |
| 手动补录和测试 | 折叠入口，用于输入 payload 或点测试码 |

当前扫码反馈：

| 反馈 | 说明 |
|---|---|
| 震动 | 扫码处理成功后震动 |
| 提示音 | 扫码处理成功后播放短音 |
| 最近扫码记录 | 显示最近识别到的 payload |
| 结果卡片 | 显示查库存、更新库存或绑定库位的结果 |

当前 payload 逻辑：

| payload | 当前行为 |
|---|---|
| `spool:<id>` | 找到耗材卷；默认查库存，显示可用重量和库位；在盘点或绑定上下文中参与写入 |
| `part:<id>` | 找到零件；默认查库存，显示数量和库位；在盘点或绑定上下文中参与写入 |
| `weight:<g>` | 自动切换到盘点称重；等待物品码；若已经有待处理物品则更新库存 |
| `location:<code>` | 进入库位等待；在绑定库位上下文中等待物品码并更新库位 |

当前仍有一个“模式上下文”：

```text
lookup      默认查库存
stocktake  盘点称重
move        绑定库位
```

快捷扫码里已经不再显示“查库存”按钮。扫物品码默认就是查库存，上方独立的“查库存”导航用于库存列表搜索和筛选。

### 查库存

查库存页只负责查看和筛选，不负责新增。

当前按钮和控件：

| 控件 | 作用 |
|---|---|
| 搜索框 | 搜索 ID、名称、材料、颜色、库位 |
| 类型筛选 | 全部 / 耗材卷 / 零件 |
| 状态筛选 | 启用 / 全部状态 / 已归档 |
| 低库存 | 只看低库存物品 |
| 合并导入 | 合并外部 JSON 快照 |
| 覆盖导入 | 用外部 JSON 快照替换本地数据 |
| 导出快照 | 导出全部本地数据 |
| 导出当前列表 | 只导出当前筛选结果 |

点击库存行会跳到“新增”页编辑该物品。

### 新增

新增页负责物品资料维护和标签生成。

当前按钮：

| 按钮 | 作用 |
|---|---|
| 新建 | 清空表单，准备创建物品 |
| 克隆 | 从当前物品复制字段，生成一个新物品草稿 |
| 归档 / 恢复 | 停用或恢复当前物品 |
| 生成 ID | 按类型和字段生成可用 ID |
| 保存物品 | 新增或更新物品 |
| 生成标签 | 为已存在物品生成 `spool:` 或 `part:` 二维码 |

当前约定：

```text
新增物品 -> 保存 -> 自动选中该物品 -> 可生成标签
```

### 流水

流水页当前只展示改动记录。

当前记录来源：

| 操作 | 写入流水 |
|---|---|
| 盘点称重更新耗材卷重量 | 是 |
| 盘点称重更新零件估算数量 | 是 |
| 绑定库位 | 是 |

## 推荐下一步精简方向

下一步建议把扫码从“模式驱动”改成“上下文驱动”。

目标状态：

```text
activeItem     当前物品
pendingAction  none / weight / location
pendingValue   重量值或库位值
```

目标操作逻辑：

| 扫码顺序 | 行为 |
|---|---|
| 扫物品码 | 查库存，并设为当前物品 |
| 扫重量码 -> 扫物品码 | 更新该物品库存 |
| 扫物品码 -> 扫重量码 | 更新该物品库存 |
| 扫库位码 -> 扫物品码 | 绑定该物品库位 |
| 扫物品码 -> 扫库位码 | 绑定该物品库位 |

目标扫码页按钮可以进一步减少为：

```text
开始扫码 / 停止 / 手电筒 / 撤销上一步 / 手动补录
```

也就是说，最终不需要“盘点称重”和“绑定库位”模式按钮。二维码标签和扫码顺序应该足够决定动作。

## 风险点

后续实现上下文驱动时，要特别注意：

- 重复扫码防抖，避免连续写两次。
- 写入后要有明确结果、震动和声音。
- 写错后的“撤销上一步”比每次弹确认更高效。
- 扫到未知物品时不能吞掉待处理重量或库位。
- 扫到新重量时应覆盖旧的待处理重量。
- 扫到新库位时应覆盖旧的待处理库位。
- Android 原生扫码以后也只应该把 payload 交给核心，不直接改库存。
