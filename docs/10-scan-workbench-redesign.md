# 扫码台重设计

## 结论

0.1.4 的快捷扫码页仍然是“模式按钮 + 状态条”的补丁模型，不适合作为长期交互基础。

下一版应把它推翻成一个统一的“扫码台”：

```text
扫到什么 -> 系统识别 -> 当前卡片展示 -> 必要时直接写入或给明确按钮
```

不要让用户先判断自己在“查库存 / 盘点称重 / 绑定库位”哪个模式里。

## 导航结构

主导航仍保留：

```text
扫码 / 库存 / 新增 / 流水
```

但含义调整：

| 导航 | 新职责 |
|---|---|
| 扫码 | 唯一扫码工作台，处理查库存、盘点、移库、出库、重新入库 |
| 库存 | 搜索、筛选、浏览列表；可打开同一个扫码台做“扫码查库存” |
| 新增 | 新增/编辑物品，生成标签 |
| 流水 | 查看写入记录，后续承载撤销 |

库存页如果加“扫码”按钮，不应另写一套扫码逻辑，只是打开同一个扫码台并带上 `intent=lookup`。

## 页面结构

扫码页从上到下：

```text
相机预览
  开始 / 停止 / 手电筒

当前卡片
  当前物品或待处理动作

主操作按钮
  只显示当前能做的动作

最近结果
  最近扫码、最近写入、撤销入口

手动补录和测试
  默认折叠
```

不再显示固定的“盘点称重 / 绑定库位”模式按钮。

## 状态模型

旧模型：

```text
mode = lookup / stocktake / move
pendingWeight
pendingItem
pendingLocation
```

新模型：

```text
currentItem     当前物品，可为空
pendingAction   none / weight / location
pendingValue    重量或库位
lastWrite       最近一次写入，用于撤销
intent          workbench / inventoryLookup
```

核心判断不看用户点了哪个模式，而看：

```text
当前扫到的标签 + 当前是否已有待处理动作 + 当前物品状态
```

0.2 的数据边界以 [12-msi-v1-and-0.2-scope.md](./12-msi-v1-and-0.2-scope.md) 为准：

```text
ItemProfile = 二维码固定档案
ItemState   = 本地当前库存状态
Transaction = 操作流水
```

有 `ItemProfile` 但没有 `ItemState` 时，表示“已建档，未入库”，不计入库存。不再引入 `unconfirmed` 状态。

## 扫码事件规则

### 扫到物品码

payload：

```text
spool:<id>
part:<id>
msi:v1;type=spool;...
msi:v1;type=part;...
msi:v1;type=other;...
```

规则：

| 当前状态 | 行为 |
|---|---|
| 没有待处理动作 | 展示物品详情，设为当前物品 |
| 待处理重量 | 对该物品写入重量/数量 |
| 待处理库位 | 对该物品绑定库位 |
| 物品已出库且待处理重量 | 重新入库，并写入新重量 |
| 物品已出库且无待处理重量 | 展示已出库详情，提示扫重量码或手动录入重量后重新入库 |
| 本地没有档案但扫到完整 `msi:v1` | 恢复固定档案，只建 `ItemProfile`，不自动入库 |

扫物品码默认就是查库存。

### 扫到重量码

payload：

```text
weight:<g>
msi:v1;type=weight;value_g=<g>
```

规则：

| 当前状态 | 行为 |
|---|---|
| 当前有在库物品 | 直接更新该物品重量/数量 |
| 当前有已出库耗材卷 | 重新入库，并写入新重量 |
| 当前无物品 | 设置待处理重量，提示继续扫物品码 |

扫重量不再“切换模式”，因为已经没有模式。

### 扫到库位码

payload：

```text
location:<code>
```

规则：

| 当前状态 | 行为 |
|---|---|
| 当前有物品 | 直接绑定库位 |
| 当前无物品 | 设置待处理库位，提示继续扫物品码 |

如果已经有待处理库位，再扫新的库位码，新的覆盖旧的。

## 当前卡片内容

### 耗材卷详情

扫码 `spool:<id>` 后展示：

```text
物品 ID
名称 / 品牌 / 材料 / 颜色
当前状态：在库 / 已出库 / 已归档
当前重量
可用重量
库位
入库时间
最近更新时间
同类耗材剩余卷数
```

同类耗材不单独依赖 `family` 字段。0.2 按固定档案字段自动统计：

```text
brand + material + color
```

`family` 不进 `msi:v1`，避免二维码变长和长期维护出错。

### 零件详情

扫码 `part:<id>` 后展示：

```text
物品 ID
名称
当前数量
库位
报警值
最近更新时间
```

零件是否需要“出库”先不放进第一版扫码台，避免把耗材卷逻辑和零件领用逻辑混在一起。

### 其他物品详情

扫码 `other` 后展示：

```text
物品 ID
名称
备注
建档日期
当前状态：未入库 / 在库 / 已出库 / 已归档
库位
最近更新时间
```

`other` 只做登记、查找、库位和状态流转，不做重量或数量余量计算。

## 出库和重新入库

出库不是归档。

建议字段：

```text
checked_out_on   出库日期，YYMMDD，空表示在库
stocked_on       本次入库日期，YYMMDD
created_at       流水精确时间，带时区
```

### 出库当前卷

在耗材卷详情卡上显示：

```text
出库当前卷
```

点击后：

```text
checked_out_on = 今天
写流水 checkout
当前卷状态变成已出库
同类剩余卷数减少
```

不删除物品，不归档物品，不改二维码。

### 重新入库

已出库卷再次扫码时：

```text
展示已出库详情
提示：扫重量码或手动录入重量后重新入库
```

重新入库必须有重量。

允许两条路径：

```text
扫已出库卷码 -> 扫 weight -> 自动重新入库
扫 weight -> 扫已出库卷码 -> 自动重新入库
```

重新入库后：

```text
checked_out_on = ""
current_wt = 新重量
stocked_on = 今天或用户自定义日期
state_updated_on = 今天
写流水 checkin
```

## 主操作按钮规则

按钮不固定展示，按当前卡片状态出现。

| 状态 | 按钮 |
|---|---|
| 相机未开 | 扫码 / 预览扫码（实验） |
| 相机已开 | 停止、手电筒 |
| 有待处理动作 | 取消待处理 |
| 物品在库且是耗材卷 | 出库当前卷 |
| 物品已出库且无重量 | 手动录入重量 |
| 最近有写入 | 撤销上一步 |
| 任意物品详情 | 编辑物品 |

不显示“查库存 / 盘点称重 / 绑定库位”模式按钮。

## 写入反馈

所有写入动作必须有明确反馈：

```text
声音
震动
结果卡片变色或高亮
流水新增
可撤销入口
```

只扫码查看，不写入时也可以轻微声音/震动，但视觉上要和“已写入”区分。

## 防错规则

- 待处理重量和待处理库位同时只能有一个。
- 新扫重量覆盖旧待处理重量。
- 新扫库位覆盖旧待处理库位。
- 扫未知物品不能清掉待处理动作。
- 已出库耗材不能直接盘点重量，必须走重新入库。
- 归档物品不能出库或重新入库，只能查看或恢复。
- 连续识别同一二维码需要防抖，避免重复写入。
- 写入后要保留最近一次变更，支持撤销。

## 实现顺序

建议不要在 0.1.4 页面上继续补按钮，按下面顺序重做：

1. 实现 `msi:v1` parser，并兼容旧短码。
2. 拆 `ItemProfile / ItemState / Transaction`。
3. 新建核心扫码 reducer：输入 payload 和当前上下文，输出 `view / write / message`。
4. 重做扫码页 UI，只保留相机、当前卡片、动态按钮。
5. 库存页扫码按钮复用同一个扫码台。
6. 增加入库、出库、重新入库、移库、撤销上一步。
7. 导出/导入 JSON 快照。
8. 发布新版 APK。

## 二维码测试工具要求

测试工具应该同时支持 `msi:v1` 推荐协议和旧短码兼容协议。

推荐协议：

```text
msi:v1;type=spool;...
msi:v1;type=part;...
msi:v1;type=other;...
msi:v1;type=location;...
msi:v1;type=weight;value_g=...
```

旧短码兼容：

```text
spool:<id>
part:<id>
weight:<g>
location:<code>
raw:<custom>
```

但必须提供“扫码台场景套码”，覆盖这些顺序：

| 场景 | 顺序 |
|---|---|
| 查耗材详情 | `spool:<id>` |
| 查零件详情 | `part:<id>` |
| 先称重再扫物品 | `weight:<g>` -> `spool:<id>` |
| 先物品再称重 | `spool:<id>` -> `weight:<g>` |
| 先库位再扫物品 | `location:<code>` -> `spool:<id>` |
| 先物品再移库 | `part:<id>` -> `location:<code>` |
| 出库当前卷 | `spool:<id>` -> app 里点“出库当前卷” |
| 出库卷重新入库 | `spool:<id>` -> `weight:<g>` |
| 先重量后重新入库 | `weight:<g>` -> `spool:<id>` |
| 同类耗材卷数 | 多个同类 `spool:<id>` |

出库和重新入库是 app 行为，不应该设计成新的二维码协议。
