# 工作室物品管理 - 最小化架构决策

## 结论

先采用 **QR Input Web Core + Capacitor Android Shell**。

- 核心：纯 Web/TypeScript 方向，业务逻辑、库存计算、同步策略和二维码协议都不依赖 Android。
- 当前最小实现：零构建静态 Web，先跑通二维码输入驱动的库存闭环。
- Android 外壳：后续用 Capacitor 包 APK，接入相机、文件系统、BLE 打印。
- 未来 Docker/Web：同一套 Web Core 放进容器，由 HTTP 服务托管即可。

## 为什么不是一开始做原生 Android

原生 Android 会让扫码、存储、打印都很快进入平台细节，后续迁移网页或 Docker 时会重复开发。当前需求的核心不是 Android UI，而是：

- 物品协议：`spool:`、`part:`、`weight:`、`location:`
- 库存计算
- 流水记录
- 本地数据库快照
- WebDAV 同步

这些都适合先放在平台无关核心里。

## 后续演进

### Phase 0：当前检测版

- 浏览器静态服务即可用。
- 用 localStorage 保存数据。
- 手动输入或复制扫码内容，模拟扫码枪/相机输入。
- `查库存 / 盘点称重 / 绑定库位` 都由二维码输入触发。
- 二维码生成器只作为 `tools/` 测试页和 `打标签` 辅助。

### Phase 1：Android APK

- 保留 `core.js` 的业务逻辑。
- 用 Capacitor 包装 `app/`。
- 用相机插件替换手动扫码输入。
- 用 Capacitor Filesystem + SQL.js 替换 localStorage。

### Phase 2：同步

- 本地数据库导出快照。
- WebDAV 定时上传/下载。
- 按 `updated_at` 合并记录。
- 单用户场景采用最新时间戳覆盖。

### Phase 3：Docker/Web

- 把同一套 `app/` 作为静态资源托管。
- 数据层切换为服务端 SQLite 或继续同步快照。
- WebDAV 同步仍可作为外置备份通道。

## 推荐框架边界

当前先不用 React/Vue/Svelte。原因是最小闭环只有库存状态机、列表、表单、二维码生成，原生 Web 足够。

等功能超过以下任一条件，再引入框架：

- 页面状态开始跨多个视图共享且难以维护。
- 需要复杂组件、权限、批量编辑、仪表盘。
- 需要长期多人维护。

届时推荐：

- UI 框架：Svelte + Vite，包体小，移动表单开发快。
- 壳：Capacitor。
- 数据核心：独立 `core/` 包，不依赖 UI。
